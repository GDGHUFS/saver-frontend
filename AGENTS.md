# Saver Frontend 작업 지침

이 저장소에서는 루트의 `openapi.json`과 사용자가 전달한 화면 요구사항을 바탕으로 포털 서비스의 프런트엔드를 개발한다.

기본 화면 골격은 [GDGHUFS/saver-wireframe](https://github.com/GDGHUFS/saver-wireframe)을 따른다. 다만 이 프로젝트는 Codex가 구현을 주도하고 사용자가 리뷰하는 방식이므로, 단기적인 화면 완성보다 이후 변경과 검증이 쉬운 코드를 우선한다. 또한 사용자는 openapi.json을 만드는 backend이므로 필요에 따라 backend에 관련한 요청이나 질문을 할 수 있다.

## 명세 우선순위

요구사항이 충돌하면 다음 순서로 판단한다.

1. 사용자가 현재 작업에서 명시한 요구사항
2. 이 `AGENTS.md`
3. 루트의 `openapi.json`
4. 기존 코드와 테스트가 나타내는 동작
5. `saver-wireframe`의 화면 구조

- 불명확한 부분은 먼저 저장소와 명세에서 근거를 찾는다.
- 합리적이고 되돌리기 쉬운 결정은 기존 패턴에 맞춰 진행하고, 보안·데이터 손실·API 계약처럼 영향이 큰 결정만 사용자에게 확인한다.
- `openapi.json`에 없는 API나 응답 필드를 추측해 구현하지 않는다. 미완성 기능은 범위가 드러나는 TODO와 비활성 상태로 남긴다.

## 기술과 코드 품질

- Vue.js 3, Vite, TypeScript를 사용하고 새 컴포넌트는 Composition API와 `<script setup lang="ts">`를 기본으로 작성한다.
- TypeScript의 엄격한 타입 검사를 유지한다.
  - `any`, `@ts-ignore`, 무검증 type assertion 등 암시성을 높이는 요소를 사용하지 않는다.
  - 외부 입력과 API 응답은 `unknown`에서 시작해 타입 가드 또는 검증 함수로 좁힌다.
  - 도메인 타입과 API DTO를 컴포넌트 안에 중복 선언하지 않는다.
- UI는 Bootstrap 5의 레이아웃, 유틸리티, 컴포넌트 클래스를 우선 사용한다.
  - 새로운 UI 라이브러리를 추가하지 않는다.
  - Bootstrap으로 표현하기 어려운 고유 디자인만 범위가 좁은 CSS로 보완한다.
  - 색상과 간격은 가능한 한 CSS 변수로 관리하고 동일한 값을 여러 컴포넌트에 하드코딩하지 않는다.
- 라우팅은 Vue Router로 관리하며 페이지 이동에 일반 `<a>` 대신 `RouterLink`를 사용한다. 외부 링크와 backend OAuth 진입점은 예외다.
- 컴포넌트는 한 가지 역할만 맡도록 구성한다. 페이지 컴포넌트는 데이터 조합과 화면 상태를 담당하고, 반복되는 표현과 상호작용은 작은 컴포넌트로 분리한다.
- 중복 제거만을 위한 성급한 추상화는 피한다. 둘 이상의 화면에서 실제로 공유되는 상태 처리나 UI 패턴만 공통화한다.

## 권장 구조와 경계

프로젝트를 처음 구성할 때는 다음 경계를 기본으로 하되, 기능이 늘어날 때 기존 구조에 맞춰 확장한다.

```text
src/
  api/          # HTTP client, endpoint 함수, API DTO와 검증
  assets/       # 전역 스타일과 정적 자산
  components/   # 둘 이상의 화면에서 재사용하는 UI
  composables/  # 재사용하는 반응형 상태와 비동기 흐름
  router/       # 라우트 정의
  views/        # 라우트 단위 페이지
  types/        # API 외 공통 도메인 타입
tests/          # 핵심 사용자 흐름과 공통 로직 테스트
```

- 한 화면에서만 쓰는 컴포넌트는 해당 기능 가까이에 둔다. 모든 코드를 무조건 전역 `components`로 올리지 않는다.
- 파일과 식별자 이름은 기존 코드의 규칙을 따른다. 새 프로젝트에서는 Vue 컴포넌트에 PascalCase, 함수와 변수에 camelCase를 사용한다.
- 환경별 값은 `VITE_` 접두사의 환경 변수로 주입하고 비밀값, 실제 토큰, 운영 URL을 소스에 하드코딩하지 않는다. 필요한 변수는 `.env.example`에 문서화한다.

## API 통신

- 모든 HTTP 요청은 `src/api/client.ts` 및 `src/api/` 아래의 endpoint 모듈에 한정한다. 컴포넌트, view, store, composable에서 `fetch`를 직접 호출하지 않는다.
- 직접 Axios를 추가하지 않는다. 별도 합의가 없다면 브라우저 기본 `fetch`를 감싼 공통 client를 사용한다.
- `window.fetch`처럼 호출 수신자에 의존할 수 있는 브라우저 네이티브 메서드를 변수나 클래스 필드에 저장할 때는 원래 객체에 바인딩한다. Chromium에서 우연히 동작하는 호출 방식에 의존하지 말고 Firefox를 포함한 브라우저 동작을 회귀 테스트로 보호한다.
- API 기본 URL, 공통 헤더, JSON 파싱, timeout/취소, 오류 변환을 공통 client에서 처리한다.
- 인증은 backend가 발급하는 HttpOnly 쿠키를 사용하므로 인증 요청에는 필요한 경우 `credentials: 'include'`를 적용한다. 토큰을 Local Storage나 JavaScript가 읽을 수 있는 쿠키에 복제하지 않는다.
- credential CORS는 프런트엔드의 scheme, host, port가 모두 일치하는 정확한 Origin을 backend와 맞춘다. 개발 서버가 다른 포트로 자동 전환되거나 `localhost` 대신 `127.0.0.1`로 열리면 별개의 Origin이므로 실행 주소와 CORS 응답 헤더를 함께 확인한다.
- OAuth authorize endpoint와 외부 로그인 화면은 `fetch`로 호출하지 않고 `<a>` 또는 `window.location`을 사용해 브라우저 전체를 이동시킨다. OAuth 제공자의 로그인 화면에서 발생하는 fetch CORS 오류를 backend API CORS 오류로 오인하지 않는다.
- OAuth `code`와 `state`를 처리하고 세션 쿠키를 발급하거나 계정 연결을 해제하는 callback은 backend가 담당한다. 카카오 Redirect URI에는 backend callback을 등록하고, backend가 처리를 마친 뒤 프런트엔드로 다시 이동시킨다.
- GET query는 `URLSearchParams`로, JSON 요청은 직렬화 가능한 명시적 DTO로 구성한다.
- 204 응답과 JSON 응답을 구분하고, 성공이 아닌 HTTP 상태를 정상 데이터로 취급하지 않는다.
- `openapi.json`을 API 계약의 기준으로 삼고 path, method, status code, request/response schema를 일치시킨다. 명세가 변경되면 관련 타입, 검증, mock과 테스트도 함께 갱신한다.
- cursor나 magic code처럼 서버가 발급한 불투명한 값은 클라이언트에서 해석하거나 변경하지 않는다.
- 예상하지 못한 API 실패를 기록할 때는 작업 이름, method/path, frontend/API Origin, 오류 종류와 상태 코드처럼 재현에 필요한 정보만 구조화한다. 쿠키, 토큰, 사용자 정보와 검증되지 않은 응답 본문은 로그에 남기지 않으며, 로그인하지 않은 사용자의 정상적인 401 같은 계약상 분기는 오류로 기록하지 않는다.

### 외부 JavaScript SDK

- 외부 SDK의 `<script>` `load` 이벤트와 SDK 내부 초기화 완료를 같은 상태로 간주하지 않는다. 스크립트 로드, SDK 초기화, 실제 리소스 로드를 구분해 실패 단계가 드러나도록 처리한다.
- Kakao Maps SDK를 `autoload=false`로 동적 로딩할 때 스크립트 직후의 `kakao.maps`는 `load(callback)`만 제공하는 부트스트랩 네임스페이스일 수 있다. 이 상태에서 `Map`, `LatLng`, `event` 같은 전체 API의 존재를 먼저 요구하지 말고 `kakao.maps.load()`를 호출한 뒤 콜백 안에서 전체 API를 검증하고 사용한다.
- 외부 SDK 로더 테스트는 완전히 초기화된 전역 객체만 미리 주입하지 않는다. 가능한 경우 실제 지연 로딩처럼 부트스트랩 객체를 먼저 제공하고 초기화 콜백에서 전체 API가 준비되는 순서를 재현한다.
- SDK 진단 로그에는 frontend Origin, 단계, 상태 코드처럼 재현에 필요한 정보만 기록한다. API 키와 키가 포함된 전체 SDK URL은 기록하지 않는다.

## 화면 상태와 UX

- API를 사용하는 각 화면은 `loading`, `empty`, `error`, `success` 상태를 명시적으로 구현한다.
- 공통 상태 표현은 재사용 가능한 컴포넌트나 composable로 통일하되, 화면별 복구 동작은 구체적으로 제공한다.
  - loading: 레이아웃이 크게 움직이지 않는 진행 상태
  - empty: 오류와 구분되는 빈 결과 안내
  - error: 사용자가 이해할 수 있는 메시지와 가능한 경우 재시도 동작
  - success: 중복 요청과 오래된 응답으로 상태가 덮이지 않도록 처리
- 작성, 수정, 삭제, 로그아웃처럼 상태를 바꾸는 동작은 처리 중 중복 제출을 막고 성공·실패 결과를 표시한다.
- 파괴적인 동작에는 확인 절차를 둔다. 계정 탈퇴나 데이터 삭제의 실패를 성공처럼 처리하지 않는다.
- 모든 페이지에서 공통 네비게이션과 일관된 최대 너비, 여백, 피드백 방식을 유지한다.
- 모바일부터 넓은 화면까지 레이아웃이 깨지지 않게 구현하며, 카드 그리드는 Bootstrap breakpoint에 따라 1열, 2열, 3열로 자연스럽게 변하도록 한다.
- 키보드만으로 주요 기능을 사용할 수 있어야 한다. 입력에는 연결된 label을, 아이콘 버튼에는 접근 가능한 이름을 제공하고, 포커스 표시를 제거하지 않는다.
- 이미지에는 의미에 맞는 대체 텍스트를 제공하고, 장식 이미지는 빈 `alt`를 사용한다.

## 테스트와 검증

- 핵심 사용자 흐름과 실패 가능성이 높은 공통 로직에는 테스트를 추가한다. 테스트에는 구현 세부사항이 아니라 테스트의 의도와 보호하는 동작이 드러나는 짧은 주석을 둔다.
- API client 테스트는 최소한 성공 응답, 대표 오류 응답, 빈 응답(204), 네트워크 실패를 다룬다.
- API client가 감싼 브라우저 네이티브 메서드는 실제 브라우저가 요구하는 호출 컨텍스트도 테스트한다. 특히 기본 `fetch`가 `Window` 수신자로 호출되는지 보호한다.
- API 화면 테스트는 loading, empty, error, success 상태와 재시도 또는 주요 사용자 동작을 검증한다.
- 검색 polling, cursor pagination, 인증 분기 같은 비동기 흐름은 완료, 실패, 취소 또는 만료 경로를 검증한다.
- endpoint 테스트는 `.env`의 API 기본 URL 유무에 따라 깨지지 않도록 전체 URL 문자열을 고정하지 말고 method와 endpoint path 계약을 검증한다. URL 조합 자체를 검증하는 테스트는 별도로 둔다.
- 버그를 수정할 때는 가능한 경우 회귀 테스트를 먼저 추가하거나 같은 변경에 포함한다.
- 특별한 사유가 없다면 작업 완료 전에 다음 검증을 실행한다.
  1. `npm run build`
  2. `npm run test`
  3. 프로젝트에 설정된 lint, typecheck, format 검사
- 실행하지 못한 검증이나 남은 경고는 완료 보고에 이유와 함께 명시한다.

## Git 작업 원칙

- 이 프로젝트는 Git으로 관리한다.
- 작업 시작과 종료 시 `git status`를 확인한다.
- 사용자의 기존 변경 사항을 보존하고, 관련 없는 파일을 되돌리거나 커밋에 섞지 않는다.
- 커밋 하나는 하나의 논리적 변경 단위로 구성한다.
- 커밋보다 큰 작업 단위는 브랜치로 관리한다(예: 메인 페이지 브랜치). 필요에 따라 브랜치 생성을 권하거나 수행할 수 있다.
- 사용자가 요청하지 않은 커밋, push, rebase, reset은 수행하지 않는다.
- 커밋 메시지는 반드시 한국어로 상세하게 작성한다.
  - 제목에는 변경 목적을 명확히 쓴다.
  - 본문에는 주요 구현 내용과 필요한 경우 검증 방법 및 영향 범위를 설명한다.

## 완료 기준

작업은 다음 조건을 충족해야 완료로 간주한다.

- 요청한 기능과 네 가지 API 화면 상태가 구현되어 있다.
- TypeScript 오류와 브라우저 콘솔 오류가 없다.
- 모바일과 데스크톱에서 핵심 레이아웃, 긴 텍스트, 빈 데이터가 깨지지 않는다.
- 관련 테스트가 추가 또는 갱신되었고 build와 test 결과가 확인되었다.
- 접근성 기본 사항과 주요 키보드 흐름을 확인했다.
- 임시 mock, 디버그 로그, 사용하지 않는 코드와 근거 없는 TODO가 남아 있지 않다.
- 변경 파일, 검증 결과, 남은 제한 사항을 완료 보고에 간결하게 정리했다.

## 화면과 기능

다음 화면을 제공해야 한다. 상세한 구현 지침이나 수정 사항은 이후 작업 지시를 통해 확인할 수 있다.

### 메인 페이지

메인 페이지는 다음 세 구역으로 나눈다.

1. 상단 네비게이션 바
2. 검색 바: Google 검색 바와 유사한, 익숙하고 단순한 검색 입력 UI
3. 서비스 카드 묶음: backend가 제공하는 서비스로 이동하거나 해당 기능을 사용할 수 있는 카드

서비스 카드는 넓은 화면에서 3열 2행으로 배치하고 화면 너비에 따라 반응형으로 열 수를 조절한다. 읽기 순서는 다음과 같다.

1. 카카오 소셜 로그인
2. 날씨 페이지
3. 기념일 페이지
4. 뉴스 페이지
5. 블로그 페이지
6. 서비스 소개와 개인정보처리방침

### 기념일 페이지

- 메인 페이지와 같은 상단 네비게이션 바를 포함한다.
- `openapi.json`의 연월별 특일 조회 API를 기준으로 구현한다.

### 뉴스 페이지

- 메인 페이지와 같은 상단 네비게이션 바를 포함한다.
- `openapi.json`의 발행자 및 최신 뉴스 API를 기준으로 구현한다.
